CostOps Quick Start Guide

This document is a consolidated quick start guide for the current CostOps application. It is intended for internal review before screenshots, formatting polish, website adaptation, or video scripting are added.

The goal of this draft is to explain what each major page does, how it is typically used, and why it matters.

  • Audience: enterprise administrators, platform teams, analytics teams, and business stakeholders
  • Structure: Command Center, Analysis, Value, and Admin
  • Current state: text-first draft ready for review before visual callouts are added

How to use this guide

Use the left navigation to jump between product areas and pages. Each page section explains what the page is for, what major sections it contains, how a user would typically work with it, and why it matters in the overall CostOps workflow.

Command Center

The Command Center is the main operational area of CostOps. It combines the summary view, recommendation workflow, personal or team execution queues, and scan orchestration.

Overview

The Overview page is the main landing page for CostOps. It gives users a high-level summary of cost exposure, savings opportunity, recommendation priority, and enterprise rollout status.

What this page is for

  • Review current estimated Snowflake spend
  • See open monthly savings opportunity
  • Track realized monthly savings
  • Identify the number of critical findings
  • Understand how recommendations are distributed by category, severity, and status
  • Spot the highest-value recommendations to review first

Main sections

Filter bar: lets users filter by status, category, team, owner, and minimum confidence. These filters update the dashboard below.

KPI summary cards: show estimated monthly spend, monthly savings opportunity, realized monthly savings, and critical findings.

Enterprise Rollout: for Enterprise accounts, this summarizes readiness, active areas, action needed, and the top blocker.

Savings Opportunity charts: provide category, severity, and status breakdowns.

Highest Value Recommendations: highlights recommendations with the largest financial impact, including ownership, confidence, due dates, and missed savings context.

Recommended user behavior

  1. Review the KPI cards
  2. Apply filters if needed
  3. Scan the charts for major problem areas
  4. Review the Highest Value Recommendations table
  5. Move into the Recommendations page for detailed action

Why this page matters

The Overview page is designed to answer one question quickly: Where should we focus first to reduce Snowflake cost and improve operational efficiency?

Recommendations

The Recommendations page is the main workflow page in CostOps. It is where users review savings opportunities, assign ownership, move recommendations through implementation stages, and track audit history.

What this page is for

  • Review the prioritized backlog of optimization opportunities
  • Filter recommendations by severity, status, category, team, owner, and confidence
  • Select a recommendation and review its detail
  • Assign ownership and due dates
  • Log SQL copy activity
  • Move a recommendation through its lifecycle from discovery to realization

Main sections

Filter bar: severity, status, category, team, owner, minimum confidence, and sort controls.

KPI summary cards: estimated monthly spend, monthly savings opportunity, realized monthly savings, and critical findings.

Recommendation summary chips: show total findings, savings opportunity, missed savings, average age, overdue items, and realized savings.

Recommendation backlog table: the main recommendation list with columns for savings, due dates, confidence, ownership, risk, effort, and status.

Selected recommendation detail: the lower detail pane updates when a row is selected.

Generated SQL or implementation guidance: appears when available and can be logged as copied.

Assignment section: owner, team, role, due date, and assignment notes.

Workflow stage: proposed, selected, accepted, implemented, and realized.

Lifecycle and audit panel: milestone information and recent audit activity for the selected recommendation.

Recommended user behavior

  1. Apply filters to narrow the backlog
  2. Select a recommendation
  3. Review the detail and expected savings
  4. Assign ownership and due date if needed
  5. Review or copy the SQL
  6. Move the recommendation through the workflow
  7. Use the audit section to confirm history and actions

Why this page matters

The Recommendations page is the operational core of CostOps. It takes the platform from simply detecting inefficiencies to actually managing the work required to reduce cost and realize savings.

My Work

The My Work page is the personal and team execution view in CostOps. It is designed to help users focus on the recommendations they are directly responsible for and manage those items as an active work queue.

What this page is for

  • View recommendations by owner, team, or role
  • Focus only on assigned items
  • See overdue work and near-term due items
  • Understand the open savings tied to a specific person or group
  • Manage recommendation detail and workflow from a narrower operational view

Main sections

Work queue filters: view by owner, team, or role, along with open-only, lane, and category controls.

Summary metrics: owned recommendations, overdue, due in 7 days, and open monthly savings.

Work queue summary: a compact view of the selected lane and related savings.

Due queue table: a quick table of urgent items with due dates and missed savings context.

Owned Queue: the main working table for the selected owner, team, or role.

Selected recommendation detail, Generated SQL, Assignment section, Workflow stage, and Lifecycle and audit detail mirror the operational actions from the Recommendations page, but within a narrower personal queue view.

Recommended user behavior

  1. Choose a view by owner, team, or role
  2. Enable open-only when focusing on unresolved work
  3. Review summary metrics and due queue
  4. Select a recommendation from the Owned Queue
  5. Review its detail and expected savings
  6. Update ownership or workflow stage as work progresses

Why this page matters

The My Work page turns CostOps into a personal or team execution tool. Instead of asking, What should the organization work on? it answers, What am I responsible for right now, and what value is attached to it?

Scan & Schedule

The Scan & Schedule page is the operational control page for CostOps analysis runs. It allows users to review scan freshness, understand scan economics, configure scan timing, and run new analysis against the Snowflake environment.

What this page is for

  • See when the last analysis run occurred
  • Understand whether the scan data is still fresh
  • Review scan credit and cost estimates
  • See how much savings was identified by the latest scan
  • Configure when scans should run
  • Run a new analysis manually
  • Review scan history over time

Main sections

Enterprise scan context banner: explains whether enterprise scan setup is fully ready and highlights blockers.

Scan Control: shows last full scan, freshness, scan credits, scan cost, identified monthly savings, and savings found per $1 of scan cost.

Enterprise environment context: production account, app instance, and linked validation environments for Enterprise setups.

Scan scheduling controls: schedule, preferred scan time, scan scope, lookback window, next scheduled scan, and the Run analysis now action.

Analysis runner explanation: describes how sample mode and Snowflake mode differ.

Analysis Runner Coverage: summarizes which source areas and rule groups are included in the scan.

Scan Run History: shows past runs, status, credits, findings, and any error messages.

Recommended user behavior

  1. Review the last scan and freshness status
  2. Check estimated scan cost and identified savings
  3. Confirm production and validation context if on Enterprise
  4. Adjust schedule, scope, or lookback window if needed
  5. Run a manual scan when fresh analysis is needed
  6. Review scan history to confirm the scan completed successfully

Why this page matters

The Scan & Schedule page answers a critical operational question: When did CostOps last analyze the environment, what did that scan cost, and how much value did it uncover?

Analysis

The Analysis section helps users understand where Snowflake cost, workload behavior, storage growth, and orchestration patterns are coming from.

Warehouses

The Warehouses page provides a focused view of Snowflake warehouse cost, utilization, queueing behavior, and resume activity. It helps users understand whether warehouse resources are appropriately sized and whether there are opportunities to reduce compute cost without creating performance risk.

What this page is for

  • Analyze warehouse-level spend
  • Compare utilization against cost
  • Identify oversized or underutilized warehouses
  • Review queueing and resume behavior
  • Support warehouse rightsizing decisions

Main sections

KPI summary cards: estimated monthly spend, monthly savings opportunity, realized monthly savings, and critical findings.

Warehouse cost trend: shows spend over time across warehouses.

Warehouse summary scatterplot: compares average utilization and weekly cost to reveal potential oversizing.

Warehouse summary table: provides warehouse-by-warehouse metrics including cost, utilization, queued seconds, and resumes.

Recommended user behavior

  1. Review overall spend and savings context
  2. Use the trend chart to identify major cost contributors
  3. Use utilization versus cost to spot possible oversizing
  4. Confirm details in the warehouse summary table
  5. Move into Recommendations if a warehouse action already exists

Why this page matters

The Warehouses page helps answer: Are we paying for more warehouse capacity than we actually need?

Workloads

The Workloads page analyzes how different query, transformation, dashboard, ingestion, task, ad hoc, and procedure workloads contribute to Snowflake cost. It helps users identify which workloads are driving spend and whether scan volume, runtime, or spill behavior suggest inefficiencies.

What this page is for

  • Compare the cost of different workloads
  • Identify expensive transformations or queries
  • Review scan volume and runtime characteristics
  • Spot workloads that may benefit from redesign or optimization
  • Support decisions around query efficiency and full-refresh patterns

Main sections

Workload cost ranking: ranks workloads by total cost and groups them by workload category.

Runtime versus scan-volume scatterplot: compares GB scanned, average runtime, and relative workload intensity.

Workload Drilldown: shows workload-level metrics such as cost, warehouse, query count, runtime, scanned volume, spill, and category.

Recommended user behavior

  1. Review the workload cost ranking to identify major spend drivers
  2. Use the scatterplot to spot inefficient patterns
  3. Confirm detail in the Workload Drilldown table
  4. Compare high-cost workloads across categories
  5. Move into Recommendations if a workload-related recommendation exists

Why this page matters

The Workloads page helps answer: Which workloads are actually driving Snowflake cost, and are they using compute efficiently?

Storage

The Storage page is designed to help users understand how Snowflake storage is being consumed and where storage-related inefficiencies may exist. It focuses on object-level cost, retention, staleness, clone behavior, and opportunities for cleanup or review.

What this page is for

  • Review total storage cost
  • Identify stale objects
  • Find high-retention objects
  • Detect clone or drop candidates
  • Compare object-level storage cost across environments and schemas
  • Support cleanup and storage-governance decisions

Main sections

Storage summary metrics: monthly storage cost, stale objects, high retention objects, and clone/drop candidates.

Storage footprint treemap: shows storage cost distributed across databases, schemas, and objects.

Ranked object cost view: highlights the highest-cost storage objects by classification.

Storage Object Review: provides detailed object-level evidence, including cost, size, last queried days, retention, access count, clone group, and classification.

Recommended user behavior

  1. Review top storage metrics
  2. Use the treemap to understand where storage is concentrated
  3. Review the ranked object chart to identify high-cost objects
  4. Inspect the Storage Object Review table for detailed evidence
  5. Move into Recommendations if storage-related actions have been generated

Why this page matters

The Storage page helps answer: What data are we still paying to keep, and do we still need it in its current form?

Tasks

The Tasks page is designed to analyze Snowflake task activity, runtime behavior, failures, and compute cost. It helps users identify tasks that run too frequently, fail too often, or consume more compute than expected.

What this page is for

  • Review task execution volume
  • Identify tasks with high failure rates
  • Compare estimated compute cost across tasks
  • Detect high-frequency task activity
  • Evaluate whether tasks are running more often than needed
  • Support orchestration and scheduling optimization

Main sections

Task summary metrics: 7-day task executions, 7-day failures, estimated compute cost, and high-frequency tasks.

Task cost ranking: ranks tasks by estimated compute cost.

Runtime versus execution scatterplot: compares run frequency, runtime, and last state behavior.

Task Review Queue: gives task-level details, including schedule, failures, credits, cost, last state, and recommendation.

Recommended user behavior

  1. Review the top summary metrics
  2. Look at the task cost ranking to identify major contributors
  3. Use the runtime versus execution chart to spot noisy or unstable tasks
  4. Review the Task Review Queue for exact task-level details
  5. Use the recommendation column to guide next actions

Why this page matters

The Tasks page helps answer: Are our recurring tasks running at the right cadence and with acceptable reliability, or are they creating unnecessary cost and churn?

Value

The Value section connects recommendations to measurable business outcomes and reporting artifacts.

Savings Realization

The Savings Realization page is designed to show how much projected value has actually been captured over time. It connects recommendations to real business outcomes by comparing projected savings, realized savings, missed savings, and open recommendations.

What this page is for

  • Compare projected opportunity with realized savings
  • Track missed savings from unresolved recommendations
  • Review the number of still-open recommendations
  • Understand savings performance by category, status, team, and ownership
  • Measure whether CostOps recommendations are producing business value

Main sections

Filter bar: status, category, team, owner, and minimum confidence.

Period selector: MTD, QTD, YTD, and Since inception.

Savings summary metrics: realized savings, projected opportunity, missed savings, and open recommendations.

Savings by category chart: compares projected, realized, and missed by category.

Ownership / team savings view: compares projected monthly savings against missed savings across teams.

Savings Lifecycle by Status: status-level savings breakdown.

Aging and Ownership Queue: unresolved recommendation view with ownership and aging context.

Recommended user behavior

  1. Choose the appropriate reporting period
  2. Apply filters as needed
  3. Review the top metrics to understand the current savings posture
  4. Use category and ownership views to identify where value is or is not being captured
  5. Review the status table to understand where recommendations are in the lifecycle
  6. Use the aging queue to follow up on open items generating missed savings

Why this page matters

The Savings Realization page answers: How much value have we actually captured, and how much are we still leaving unrealized?

Reports

The Reports page is the primary reporting and export surface in CostOps. It is designed to translate recommendations, savings, ownership, scan activity, and enterprise readiness into structured reporting that can be used by finance, leadership, platform teams, and administrators.

What this page is for

  • Generate filtered reports across recommendations, savings, lifecycle, ownership, and enterprise readiness
  • Compare projected, realized, and missed savings
  • Prepare finance-friendly reporting packets
  • Review enterprise rollout context alongside cost outcomes
  • Export reporting views for sharing or review

Main sections

Enterprise reporting context banner: explains enterprise rollout context and blockers that affect reporting readiness.

Report configuration controls: report type, time range, category, team, owner, status, severity, report detail, and download format.

Supported report types: comprehensive finance packet, executive ROI summary, savings by team, savings by category, recommendation lifecycle, unresolved opportunity, owner accountability, users and roles, scan ROI history, enterprise readiness, enterprise config audit trail, recommendation backlog, recommendation audit trail, and custom report.

KPI summary cards: realized savings, projected savings, annualized opportunity, unresolved missed savings, scan ROI, and net monthly benefit.

Executive Summary: plain-language narrative for the selected report.

Enterprise Context: production account, region, app instance, billing scope, and linked validation environments.

Executive ROI Summary and additional report sections depending on the selected report type.

Recommended user behavior

  1. Choose the report type that matches the audience or objective
  2. Set the appropriate time range
  3. Apply category, team, owner, status, and severity filters
  4. Choose the desired detail level
  5. Review the KPI cards and Executive Summary
  6. Review the detailed report sections
  7. Download the report if a shareable output is needed

Why this page matters

The Reports page answers: How do we turn analysis and recommendations into a clear, shareable story about value, ownership, and progress?

RBAC status and default role: high-level posture and fallback role behavior.

Role onboarding path: supports map existing roles or use recommended roles depending on customer maturity.

Mapping action controls: edit, remove, add, and save mappings.

Source role mapping editor: source role, CostOps role, scope, and mapping status.

Why this page matters

The RBAC Mapping page answers: Who should be able to administer, operate, and view CostOps, and how does that map back to our Snowflake or identity access model?

Rule Thresholds: warehouse and task thresholds that shape recommendation logic.
